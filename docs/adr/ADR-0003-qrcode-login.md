# ADR-0003: 扫码登录模式

**状态**: 已采纳  
**日期**: 2026-04-29  
**决策者**: @qingqian  

## 背景

B 站等现代 SPA 网站的登录表单在 Modal 弹窗中动态渲染，且支持多种登录方式（手机验证码、扫码、账号密码）。账号密码自动登录面临：

1. **表单定位困难**：SPA 弹窗中的 input 字段可能在点击"登录"按钮后才渲染到 DOM
2. **验证码/滑块**：频繁触发风控，自动识别成本高
3. **多步骤流程**：手机验证码需要短信中转，无法全自动

## 决策

在原有"账号密码自动登录"（菜单 3）之外，新增**"扫码登录"模式**（菜单 4）：

### 流程
```
用户选择"扫码登录" → 浏览器打开目标页面
  → 自动点击"登录"按钮触发 Modal
  → Modal 默认显示二维码（B 站行为）
  → 打印提示，等待用户用手机 App 扫码
  → 轮询检测：URL 跳转 | 用户头像出现 | Modal 关闭
  → 检测到登录成功 → 捕捉会话 → 保存
```

### 检测成功标准（三个维度取任意一个）
1. 页面 URL 发生变化（登录后跳转）
2. `.bili-avatar` / `.user-avatar` 等用户元素出现
3. Modal 遮罩层（`.bili-mini-mask` / `.modal`）消失

### 轮询间隔：2 秒（`LOGIN_SUCCESS_POLL_MS`）  
### 超时：5 分钟（`MANUAL_LOGIN_TIMEOUT_MS`）

## 备选方案

1. **仅提供手动登录守卫（AuthGuard.manualLogin）**：依赖用户点"登录"按钮，无二维码引导
2. **自动化 WBI 签名 + API 直接登录**：需完整逆向 B 站登录协议，维护成本极高

## 后果

### 正面
- 扫码登录对目标站点的侵入性最低，无需逆向表单和密码加密逻辑
- 一次扫码获取的 Cookie/SESSDATA 可持久化复用
- 成功率远高于自动填表

### 负面
- 需要用户手动扫码，无法全无人值守
- 二维码有效期较短（约 3-5 分钟）
