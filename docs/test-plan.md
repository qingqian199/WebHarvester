# WebHarvester 实测方案（安全优先版）

**核心理念**：测试不能危害被测系统，也不能危害目标站点。所有测试必须可逆、可控、可审计。

---

## 一、安全红线（违反则立即终止测试）

| 红线 | 说明 |
|------|------|
| 🚫 禁止高频请求 | 同一站点两次请求间隔 ≥ 3 秒。不准用循环/并发压目标站点 |
| 🚫 禁止登录态测试 | 不使用真实账号密码。只测公开接口。登录态用预存的 session 文件 |
| 🚫 禁止生产数据落地 | 测试产生的 `output/` 和 `sessions/` 数据测试结束后清理 |
| 🚫 禁止外部压测 | 不对任何目标站点做并发/压力测试 |
| ✅ 安全测试范围 | 优先测本地（签名器/中间件/适配器），再测公开只读接口 |

---

## 二、测试目标（安全约束下）

| 类别 | 目标 | 安全约束 |
|------|------|----------|
| 功能完整性 | 核心模块（签名器/中间件/适配器/MCP）功能正确 | 不依赖外部站点 |
| 质量基线 | 单元测试通过率 | 全本地执行，无外部请求 |
| 性能基准 | 本地计算性能（指纹/签名/内存） | 用 `scripts/benchmark.ts`，纯本地 |
| 安全验证 | 确认项目自身的安全机制（错误处理/限流/降级） | 不攻击任何外部系统 |

---

## 三、测试环境

| 项目 | 规格 |

## 四、实测阶段

### 阶段 0：安全准备（5 分钟）

```bash
# 确保不会误伤目标站点
echo "checking config..."
grep -r "concurrency\|batch\|mass" src/config/ --include="*.ts" -l
# 确认限流配置存在
grep "rateLimit\|delay\|throttle" src/config/ --include="*.ts" -l
```

**通过标准**：限流/延迟配置存在且默认值安全。

---

### 阶段 1：本地全量测试（30 分钟，零外部请求）

此阶段**完全不访问任何外部网站**，只测项目自身代码。

```bash
# 1. 类型检查
npx tsc --noEmit
# 预期：0 errors

# 2. 运行所有安全测试（排除 E2E 和网络依赖）
bun test --timeout 30000 --filter "src/" 2>&1 | tee test-report.log

# 3. 验证核心模块
bun test src/signer/__tests__/ src/services/__tests__/capture-integration-service.test.ts --timeout 10000
# 预期：16/16 pass

# 4. 验证后端测试
bun test backend/tests/ --timeout 10000
# 预期：27/27 pass
```

**安全保证**：这些测试全部使用 mock/fixture，不发送任何 HTTP 请求。

**通过标准**：
- tsc 0 errors
- 签名器测试 16/16 pass
- 后端测试 27/27 pass
- HAR 解析测试 4/4 pass（fixture 文件测试）

### 阶段 2：本地性能基准（10 分钟，纯本地计算）

```bash
# 运行内置基准测试（本地 CPU/内存，不碰网络）
bun run scripts/benchmark.ts --runs=50
```

| 指标 | 安全原因 | 预期值 |
|------|----------|--------|
| 指纹生成 (PC/iPhone/Android) | 纯内存计算 | < 0.1ms |
| 米游社 DS 签名 | 纯本地哈希 | < 0.5ms |
| RSS 内存 | 读取 `process.memoryUsage()` | < 200MB |

**通过标准**：脚本正常运行退出，所有指标在预期范围内。

---

### 阶段 3：爬虫功能验证（节制版，1 小时）

**安全规则**：
- 每个站点 **仅 1 次请求**
- 请求间隔 ≥ 3 秒
- 只请求**公开只读数据**（列表页、搜索页）
- 不触发写入操作（不登录、不发帖、不评论）
- 使用 CLI 菜单手动操作（比自动化脚本更可控）

| 站点 | 测什么 | 请求目标 | 安全控制 |
|------|--------|----------|----------|
| **米游社** | DS 签名是否正确 | 公开帖子列表 | 1 次 GET，无 cookie |
| **Bilibili** | WBI 签名是否正确 | 公开视频页 | 1 次 GET，无 cookie |
| **知乎** | 热榜接口 | 知乎热榜（无需登录） | 1 次 GET |
| **百度学术** | SSR 论文提取 | 公开搜索页 | 1 次 GET |
| **通用浏览器** | MCP/CDP 回退 | 任意公开页面（如 example.com） | 仅 1 页 |

**禁止测试**：
- ❌ 抖音（a_bogus 签名需浏览器运行时，有反爬风险）
- ❌ BOSS 直聘（需要登录 + zp_token，触发风控）
- ❌ 小红书（用户明确要求暂停）
- ❌ 任何需要登录的接口

**通过标准**：至少 4/5 站点返回有效数据。失败的必须有明确错误码。

### 阶段 5：安全机制验证（15 分钟）

测试项目自身的安全/容错能力：

| 场景 | 操作 | 预期 |
|------|------|------|
| **错误码覆盖** | 检查 `src/core/error/ErrorCode.ts` | ≥ 35 个错误码 |


**通过标准**：错误码 ≥ 35，无硬编码凭证，路径穿越和 URL 校验生效。

## 五、绝对不做的测试

| 禁止项 | 原因 | 替代方案 |
|--------|------|----------|
| 对目标站点并发压测 | 触发反爬，被封 IP | 单次请求，间隔 ≥ 3s |
| 使用真实账号登录 | 泄露凭证，违反 ToS | 使用预存的 session 文件 |
| 自动化批量采集 | 给目标站点造成负载 | 每次手动触发 1 条 |
| 抖音/BOSS/小红书 | 签名复杂/用户要求暂停 | 记录为已知限制，不测试 |
| 7×24 稳定性测试 | 需要持续对外请求 | 仅测本地 Mock 模式循环 |

## 八、测试退出标准

任何时候出现以下情况，立即停止并回滚：

```
1. 目标站点返回 429/403/5xx —— 立即停止对该站点的所有请求
2. npm start 崩溃或 hang —— 记录日志，停止测试
3. 发现代码向目标站点发送了未预期的请求 —— 立即断网分析
4. 测试时间超过 4 小时 —— 保存已有结果，下次继续
```
