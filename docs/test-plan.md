# WebHarvester 实测方案（可执行版）

基于全景报告，验证已有功能、复现失败场景、补齐性能数据，输出投产结论。

---

## 一、测试目标

| 类别 | 目标 | 判定标准 |
|------|------|----------|
| 功能完整性 | 8 个特化爬虫 + 通用浏览器能产出数据 | 每个站点至少一次成功采集 |
| 质量基线 | 确认 37 个失败测试是否阻塞 | 核心模块测试通过率 100% |
| 性能基准 | 补齐 HTTP 延迟（内置 benchmark 已有指纹/签名/内存） | P95 < 500ms |
| 生产就绪度 | "是否可上线"明确结论 | 无 P0 阻塞项 |

---

## 二、测试环境

| 项目 | 规格 |
|------|------|
| 硬件 | 任意（本地开发机即可） |
| OS | Windows / macOS / Linux 均可 |
| 运行时 | Bun 1.3+（项目锁定 `.node-version`） |
| 浏览器 | 自动安装：`npx playwright install chromium` |
| 网络 | 需要公网出口访问目标站点。部分站点（BOSS/抖音）可能需要代理 |
| 被测版本 | `master` 最新 commit |

---

## 三、实测阶段

### 阶段 1：环境准备与冒烟测试（15 分钟）

```bash
# 1. 克隆 & 安装
git clone <repo-url>
cd WebHarvester
npm ci
npx playwright install chromium

# 2. 确认 TypeScript 编译通过（0 errors）
npx tsc --noEmit

# 3. 冒烟：启动 5 秒不崩溃
timeout 5 npm start
# 预期输出：WebHarvester CLI 菜单显示，无报错

# 4. 健康检查（另开终端）
curl http://localhost:3000/health
# 预期返回: {"status":"ok","version":"1.2.0",...}
```

**通过标准**：tsc 0 errors + 启动无异常 + `/health` 返回 200。

---

### 阶段 2：测试基线验证（30 分钟）

```bash
# 运行所有 src/ 下的测试（排除 E2E 和网络依赖场景）
bun test --timeout 30000 --filter "src/" 2>&1 | tee test-report.log
```

**关注点**：

| 测试集 | 预期 | 失败容忍 |
|--------|------|----------|
| capture-integration-service | 4/4 pass | 不可容忍 |
| signer-registry + wbi-key-manager | 16/16 pass | 不可容忍 |
| MiyousheCrawler | 全部 pass | 不可容忍 |
| ConsoleLogger / FileSessionManager | pass | 不可容忍 |
| backend/tests/ | 27/27 pass | 不可容忍 |
| BaiduScholarCrawler | 可能超时 | 可容忍（网络依赖） |
| BossSessionProvider / CrawlerIntegration | 可能 flaky | 记录为 known flaky |
| CLI E2E / Web E2E | 可能失败 | 可容忍（交互环境依赖） |

**通过标准**：核心模块测试（前三行）全部通过。其余失败记录原因。

---

### 阶段 3：功能实测（2 小时）

分两组执行：**CLI 菜单路径** 和 **Web UI 路径**。只需一组通过即可。

#### 方式 A：CLI 菜单（推荐）

```bash
npm start
# 进入交互菜单，按以下路径操作：
```

| 站点 | 菜单路径 | 验证点 |
|------|----------|--------|
| **米游社** | 特化站点采集 → miyoushe → 输入帖子 URL | DS 签名自动生成，返回帖子内容 |
| **Bilibili** | 特化站点采集 → bilibili → 输入视频 URL | WBI 签名，返回视频信息 + 评论 |
| **知乎** | 特化站点采集 → zhihu → 输入问题 URL | 热榜数据，评论可能为空（记录） |
| **百度学术** | 特化站点采集 → baidu_scholar → 输入关键词 | 返回论文列表（标题/作者/摘要） |
| **通用浏览器** | 通用浏览器采集 → 输入任意 URL → 可选设备 | MCP 或 CDP 打开页面，返回标题/截图 |

#### 方式 B：Web UI

```bash
# 另开终端启动 Web 面板
npm start
# 浏览器打开 http://localhost:3000
```

Web UI → 采集中心 → 选择站点 → 输入参数 → 开始采集。

**通过标准**：每个站点至少成功返回一次结构化数据（`output/{site}-{timestamp}/data.json`）。

---

### 阶段 4：性能基准（10 分钟）

```bash
# 运行内置基准测试
bun run scripts/benchmark.ts --runs=50
```

预期输出的各项指标均在亚毫秒级：
- 指纹生成：< 0.1ms
- DS 签名：< 0.5ms
- 内存：RSS < 200MB

**通过标准**：脚本正常退出，平均耗时 < 1ms。

---

### 阶段 5：端到端场景验证（30 分钟）

| 场景 | 操作 | 验证 |
|------|------|------|
| **采集 → 导出** | 特化站点采集 B 站 → 菜单选"导出评论" | 生成 Excel 文件 |
| **移动端模拟** | 通用浏览器采集 → 选 iPhone 模式 → 打开任意 H5 页面 | UA 和视口为移动端 |
| **异常恢复** | 先关掉 Chrome 再启动采集 | 自动回退到 MCP 模式，不崩溃 |

## 六、注意事项

- **不要爬过量**：每个站点采集 1-2 次即可，不要持续压测
- **只采公开数据**：无需登录即可访问的内容
- **首次失败先查网络**：很多"失败"是代理/防火墙导致，不是代码问题
- **时间预算**：总耗时约 **3-4 小时**（包含环境搭建和文档填写）
